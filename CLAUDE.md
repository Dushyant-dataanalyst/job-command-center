## Project

## Key facts
- **4-Strategy** = 4 voters (Inna, Pham, Cianni, Unger) each vote BUY/STRONG_BUY/WATCH/NO_SIGNAL per stock; weighted consensus via `voter_weights.json`. Logic in `equity_scan_core.py` is a *reconstruction* from one-line hints, not a verified spec.
- Every JSON feed at repo root is CI-generated; dashboard fetches client-side. `vercel.json` needs a no-cache route per new JSON file.
- Timestamps: always use `ist_time.now_ist_str()` ‚Äî bare `datetime.now()` on CI runners is UTC mislabeled as IST (past bug).
- All JSON writes: `write_text(..., encoding="utf-8")` ‚Äî Windows cp1252 default corrupted files before.
- Emoji in Python source/print crashes Windows cp1252 console ‚Äî use `[OK]`/`[WARNING]` in scripts.
- Main cron `nse_fo_refresh.yml` ‚Äî two jobs: `gate` (runs on every trigger, decides `should_run` from real IST time + `logs/run_history.json`'s last entry, not from which cron string fired ‚Äî needed since `github.event.schedule` doesn't exist for the external cron-job.org `workflow_dispatch` trigger) ‚Üí `fo-refresh` (`needs: gate`, `if: should_run`, the actual 7-script + once-daily-gated chain). **Market hours (Mon-Fri 07:30-16:25 IST): every trigger runs. Off-hours/weekends: throttled to ~hourly (55-min gap check in `should_run_full_refresh.py`) regardless of how often the trigger itself fires** ‚Äî cron-job.org's external ping runs every 5 min around the clock unrestricted; the throttling happens server-side, not by restricting cron-job.org's own schedule. Manual `workflow_dispatch` always runs, never throttled. Heartbeat (healthchecks.io) lives in `gate` so it fires on every trigger even when `fo-refresh` is skipped ‚Äî the dead-man's-switch checks "is the trigger alive", independent of the cadence decision. Telegram alert (INFO success / CRITICAL failure) via `alerts.py`'s `send_alert()` only fires when `fo-refresh` actually runs, not on throttled/skipped triggers.
- `concurrency.cancel-in-progress` is `false` (not `true`) ‚Äî `logs/run_history.json` changes every execution and forces a commit, so commit-count is a reliable execution-count proxy; with `true`, most runs got killed by the next 5-min tick before reaching the commit step. Don't re-flip this without re-checking commit cadence first.
- GitHub's native `schedule:` trigger is documented best-effort and can silently skip ticks ‚Äî known issue, don't re-diagnose from scratch. Mitigated (not fixed) by the external cron-job.org trigger hitting `workflow_dispatch` independently.
- Kite Connect (`kite_fallback.py`): daily manual token via `kite_auth_refresh.yml` workflow_dispatch; session in `kite_session.json` (expires daily). Two roles: (a) quote-price *fallback* when yfinance fails; (b) real portfolio reads ‚Äî `get_holdings/get_positions/get_margins/get_trades/get_orders`. `kite_portfolio_refresh.py` writes `kite_portfolio.json` (real holdings/margin/reconciliation vs manual tracking, plus appends real fills to `kite_trade_history.json` ‚Äî Kite's API is day-scoped only, no historical trade retrieval, so this is the sole way to accumulate real trade history) each run, failing closed+graceful (`session_live:false`) when token stale; dashboard "Real Portfolio" panel in My Positions (My Positions tab de-dupes against it via `KITE_HELD_SYMBOLS` ‚Äî a stock shown there with real Kite data doesn't also get a conflicting manual card). **No orders** (SEBI static IP). `kite_portfolio.json`/`kite_trade_history.json` are *public* static feeds (user accepted holdings exposure). **Phase 2 done and enabled**: `market_data.get_ohlcv()` tries Kite historical first (`USE_KITE_HISTORICAL=true` in the workflow env) when a live session + instrument-token resolve, falls back to yfinance automatically otherwise ‚Äî wired into `refresh_fo_cloud.py`/`equity_scan_core.py`/`market_regime_core.py`/`sector_rotation_core.py`. Verified live: candles match yfinance exactly on close/volume; fixed a real premium-formula bug in the process (`_premium_estimate` was scoring favorable ITM moves as losses ‚Äî no intrinsic-value floor).
- `daily-jobs.yml` + `fetch_jobs.py` = unrelated job-search feature in same repo; don't touch.

## Master Brief (Downloads/NSE_Trading_System_Master_Brief.md)
Locked config: risk 1% MIS / 1.5% CNC, min R:R 1.5/2.0, max 20%/position, kill switches 2% daily / 5% weekly, DRY_RUN default ‚Äî all in `nse-trading-bot/config.py`. These 2%/5% kill switches are Module 3 (not built), and are intentionally distinct from the dashboard's own existing circuit breaker (`DAILY_MAX_LOSS_PCT`/`WEEKLY_MAX_LOSS_PCT` = -3%/-8% in `nse_live_dashboard.html`) ‚Äî user explicitly said keep the dashboard one as-is, don't reconcile the two. Order code needs static whitelisted IP (SEBI Apr 2026) ‚Äî cannot run on GH Actions/Vercel. Build gates: audit ‚Üí approval ‚Üí backtest FIRST ‚Üí executor ‚Üí filters ‚Üí safety. System B (quality-growth screen) = separate folder, never orders.
- **Module 2 (`nse-trading-bot/trade_filters.py`) built, filters 1-4 only**, per brief's own "build 1-4, pause, then 5-6": market regime (reads `market_regime.json`'s avoid-list), time window (first/last 15min + optional lunch), loss streak breaker (3 consecutive losses), R:R rejection (1.5 MIS / 2.0 CNC). Standalone/testable via `python trade_filters.py` ‚Äî **not yet wired into any trade-opening code** (e.g. `trade_brain.py`'s `_open_trade()`) since that would change existing paper-trading behavior; needs explicit user go-ahead first. Filters 5 (correlation/exposure) and 6 (liquidity/spread) not built ‚Äî paused per the brief.
