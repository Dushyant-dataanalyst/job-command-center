# NSE Trading Dashboard — Project Memory

## Project
**NSE 4-Strategy Dashboard** — single-file `nse_live_dashboard.html` (~2,600 lines, no framework), Python scripts in `nse-trading-bot/`, GitHub Actions cron, Vercel static + serverless. Repo: `Dushyant-dataanalyst/job-command-center`. Owner: Dushyant, ₹1Cr goal in 2 years, ~₹1.5L MIS capital.

## Key facts
- **4-Strategy** = 4 voters (Inna, Pham, Cianni, Unger) each vote BUY/STRONG_BUY/WATCH/NO_SIGNAL per stock; weighted consensus via `voter_weights.json`. Logic in `equity_scan_core.py` is a *reconstruction* from one-line hints, not a verified spec.
- Every JSON feed at repo root is CI-generated; dashboard fetches client-side. `vercel.json` needs a no-cache route per new JSON file.
- Timestamps: always use `ist_time.now_ist_str()` — bare `datetime.now()` on CI runners is UTC mislabeled as IST (past bug).
- All JSON writes: `write_text(..., encoding="utf-8")` — Windows cp1252 default corrupted files before.
- Emoji in Python source/print crashes Windows cp1252 console — use `[OK]`/`[WARNING]` in scripts.
- **Repo is PUBLIC as of 06-Jul-2026 (was private) — this is deliberate, planned to REVERT TO PRIVATE ~01-Aug-2026.** Cause: hit 100% of the 2,000 free Actions minutes/month days into the cycle (private-repo quota resets 01-Aug-2026) — the external cron-job.org trigger fired `workflow_dispatch` every 5 min 24/7 unrestricted, and GitHub bills every job invocation rounded UP to a full minute, so even the `gate` job's fast skip decision off-hours cost a full billed minute, ~180×/day on top of market-hours runs. Rather than throttle further, went public — public repos get free/unlimited Actions minutes on standard runners, which fully removes the billing constraint. **Accepted tradeoff while public: `kite_session.json` is committed daily by `kite_auth_refresh.yml` and holds a live Kite `access_token` — anyone can read it and call Kite read-endpoints (holdings/positions/margins/trades/orders) with the user's real account until the repo reverts to private.** A reminder is scheduled for 01-Aug-2026 (also see project memory) — if this project comes up in a session at or after that date and the repo is still public, flag it proactively; the reason to stay public no longer applies once the original quota has reset.
- `nse_fo_refresh.yml` — back to the original two-job `gate`→`fo-refresh` structure with the full `schedule:` (5-min market hours, hourly off-hours, once-daily 9pm prep) + `workflow_dispatch`, restored 06-Jul-2026 once the repo went public. The off-hours throttle (`nse-trading-bot/should_run_full_refresh.py`, 55-min gap check) is kept as a **product decision** now, not cost control — 5-min granularity off-hours has no signal value (market's closed) and only adds commit/Vercel-redeploy noise. Heartbeat (healthchecks.io) lives in `gate` so it fires on every trigger even when `fo-refresh` is skipped. **If the repo ever reverts to private while cron-job.org's unrestricted external trigger is still active, this WILL recreate the original billing crisis** — before that revert (or before relying on this design past 01-Aug-2026), move the run/skip decision off GitHub Actions entirely (e.g. a Vercel serverless function gating before it ever calls `workflow_dispatch`) so a "decide to skip" never costs a billed Actions minute again.
- `concurrency.cancel-in-progress` is `false` (not `true`) — `logs/run_history.json` changes every execution and forces a commit, so commit-count is a reliable execution-count proxy; with `true`, most runs got killed by the next 5-min tick before reaching the commit step. Don't re-flip this without re-checking commit cadence first.
- GitHub's native `schedule:` trigger is documented best-effort and can silently skip ticks — known issue, don't re-diagnose from scratch. Mitigated (not fixed) by the external cron-job.org trigger hitting `workflow_dispatch` independently (now harmless cost-wise while the repo is public).
- Kite Connect (`kite_fallback.py`): daily manual token via `kite_auth_refresh.yml` workflow_dispatch; session in `kite_session.json` (expires daily). Two roles: (a) quote-price *fallback* when yfinance fails; (b) real portfolio reads — `get_holdings/get_positions/get_margins/get_trades/get_orders`. `kite_portfolio_refresh.py` writes `kite_portfolio.json` (real holdings/margin/reconciliation vs manual tracking, plus appends real fills to `kite_trade_history.json` — Kite's API is day-scoped only, no historical trade retrieval, so this is the sole way to accumulate real trade history) each run, failing closed+graceful (`session_live:false`) when token stale; dashboard "Real Portfolio" panel in My Positions (My Positions tab de-dupes against it via `KITE_HELD_SYMBOLS` — a stock shown there with real Kite data doesn't also get a conflicting manual card). **No orders** (SEBI static IP). `kite_portfolio.json`/`kite_trade_history.json` are *public* static feeds (user accepted holdings exposure). **Phase 2 done and enabled**: `market_data.get_ohlcv()` tries Kite historical first (`USE_KITE_HISTORICAL=true` in the workflow env) when a live session + instrument-token resolve, falls back to yfinance automatically otherwise — wired into `refresh_fo_cloud.py`/`equity_scan_core.py`/`market_regime_core.py`/`sector_rotation_core.py`. Verified live: candles match yfinance exactly on close/volume; fixed a real premium-formula bug in the process (`_premium_estimate` was scoring favorable ITM moves as losses — no intrinsic-value floor).
- **Telegram policy (user request 06-Jul-2026): NO routine per-run pings.** `signal_alerts.py` (end of each refresh) batches ONLY new actionable events into one message — F&O consensus flipping to BUY_CE/BUY_PE, new equity STRONG_BUYs, paper-trade closes, real tracked positions crossing SL/T1/T2, and **`_kite_fo_watch_events`: real held Kite F&O positions where the underlying signal flips AGAINST the held direction** (hold a CE, consensus goes WAIT/BUY_PE = exit-review ping; exact SL/T1 alerts aren't possible server-side since the user's F&O premium targets live only in browser localStorage, never synced). All deduped across runs via `logs/alert_state.json` (committed CI state; deliberately NOT a dashboard feed, so not in vercel.json/SCHEMA). NOTE: Kite `/positions` occasionally returns an empty `net` post-market even with a live session (seen 06-Jul-2026 16:02, back by 16:26) — the F&O watch skips its state-prune when positions[] is empty so a blip doesn't forget a still-open position, and the dashboard's F&O section just hides for that one refresh. `record_run_outcome.py` alerts CRITICAL on step failures only — its INFO-success-every-run ping was removed; "pipeline went silent" is healthchecks.io's job. Don't re-add success pings.
- **Twin implementations — the premium model exists in TWO places that must stay identical**: `refresh_fo_cloud.py::_premium_estimate()` (Python) and `nse_live_dashboard.html::_estimatePremium()` (JS, My F&O Positions panel). Guarded by `nse-trading-bot/check_premium_parity.py` + `.github/workflows/premium_parity.yml` (fires on push when either file changes; extracts the real JS, runs it in Node, compares 122 cases, asserts premium ≥ intrinsic in both). Changing one side without the other = CI failure, by design — this exact drift misprized a real position 04–06 Jul 2026 (ITM winner shown as loser). Known smaller twins not yet guarded: `_daysToExpiry` (JS `Math.ceil`) vs Python's `.days` floor — ±1 day premium wobble, cosmetic; the workflow's inline `pip install` pins vs root `requirements.txt`.
- New JSON feed checklist (3 places, all or it silently breaks): workflow `git add` list + `validate_json_outputs.py` SCHEMA + `vercel.json` (builds AND no-cache route — the `/(.*)`  catch-all otherwise serves the dashboard HTML instead of the JSON). `kite_trade_history.json` was missing from vercel.json until 06-Jul-2026.
- `daily-jobs.yml` + `fetch_jobs.py` = unrelated job-search feature in same repo; don't touch.
