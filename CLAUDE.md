# NSE Trading Dashboard — Project Memory

## Project
**NSE 4-Strategy Dashboard** — single-file `nse_live_dashboard.html` (~2,600 lines, no framework), Python scripts in `nse-trading-bot/`, GitHub Actions cron, Vercel static + serverless. Repo: `Dushyant-dataanalyst/job-command-center`. Owner: Dushyant, ₹1Cr goal in 2 years, ~₹1.5L MIS capital.

## Key facts
- **4-Strategy** = 4 voters (Inna, Pham, Cianni, Unger) each vote BUY/STRONG_BUY/WATCH/NO_SIGNAL per stock; weighted consensus via `voter_weights.json`. Logic in `equity_scan_core.py` is a *reconstruction* from one-line hints, not a verified spec.
- Every JSON feed at repo root is CI-generated; dashboard fetches client-side. `vercel.json` needs a no-cache route per new JSON file.
- Timestamps: always use `ist_time.now_ist_str()` — bare `datetime.now()` on CI runners is UTC mislabeled as IST (past bug).
- All JSON writes: `write_text(..., encoding="utf-8")` — Windows cp1252 default corrupted files before.
- Emoji in Python source/print crashes Windows cp1252 console — use `[OK]`/`[WARNING]` in scripts.
- **Repo is PUBLIC as of 06-Jul-2026 (was private) — this is deliberate, planned to REVERT TO PRIVATE ~01-Aug-2026.** Cause: hit 100% of the 2,000 free Actions minutes/month days into the cycle (private-repo quota resets 01-Aug-2026) — the external cron-job.org trigger fired `workflow_dispatch` every 5 min 24/7 unrestricted, and GitHub bills every job invocation rounded UP to a full minute, so even the `gate` job's fast skip decision off-hours cost a full billed minute, ~180×/day on top of market-hours runs. Rather than throttle further, went public — public repos get free/unlimited Actions minutes on standard runners, which fully removes the billing constraint. **Accepted tradeoff while public: `kite_session.json` is committed daily by `kite_auth_refresh.yml` and holds a live Kite `access_token` — anyone can read it and call Kite read-endpoints (holdings/positions/margins/trades/orders) with the user's real account until the repo reverts to private.** A reminder is scheduled for 01-Aug-2026 (also see project memory) — if this project comes up in a session at or after that date and the repo is still public, flag it proactively; the reason to stay public no longer applies once the original quota has reset.
- `nse_fo_refresh.yml` — back to the original two-job `gate`→`fo-refresh` structure with the full `schedule:` (5-min market hours, hourly off-hours, once-daily 9pm prep) + `workflow_dispatch`, restored 06-Jul-2026 once the repo went public. The off-hours throttle (`nse-trading-bot/should_run_full_refresh.py`, 55-min gap check) is kept as a **product decision** now, not cost control — 5-min granularity off-hours has no signal value (market's closed) and only adds commit/Vercel-redeploy noise. Heartbeat (healthchecks.io) lives in `gate` so it fires on every trigger even when `fo-refresh` is skipped. **If the repo ever reverts to private while cron-job.org's unrestricted external trigger is still active, this WILL recreate the original billing crisis** — before that revert (or before relying on this design past 01-Aug-2026), move the run/skip decision off GitHub Actions entirely (e.g. a Vercel serverless function gating before it ever calls `workflow_dispatch`) so a "decide to skip" never costs a billed Actions minute again.
- `concurrency.cancel-in-progress` is `false` (not `true`) — `logs/run_history.json` changes every execution and forces a commit, so commit-count is a reliable execution-count proxy; with `true`, most runs got killed by the next 5-min tick before reaching the commit step. Don't re-flip this without re-checking commit cadence first.

- **Module 2 (`nse-trading-bot/trade_filters.py`) built, filters 1-4 only**, per brief's own "build 1-4, pause, then 5-6": market regime (reads `market_regime.json`'s avoid-list), time window (first/last 15min + optional lunch), loss streak breaker (3 consecutive losses), R:R rejection (1.5 MIS / 2.0 CNC). Standalone/testable via `python trade_filters.py` — **not yet wired into any trade-opening code** (e.g. `trade_brain.py`'s `_open_trade()`) since that would change existing paper-trading behavior; needs explicit user go-ahead first. Filters 5 (correlation/exposure) and 6 (liquidity/spread) not built — paused per the brief.
- **Gate 2 (`nse-trading-bot/backtest.py`) built 07-Jul-2026** — replays the live 4-voter equity logic (imports `_strategy_inna/_pham/_cianni/_unger` directly from `equity_scan_core.py`, zero drift risk vs. what's actually live) against 3 years of real yfinance daily data across the 46-stock universe. Standalone, read-only, on-demand (`python backtest.py`, not wired into any CI cron, no dashboard feed, deliberately not in vercel.json/SCHEMA). Disclosed simplifications baked into every output: flat equal voter weights throughout (no retroactive replay of `voter_weights_refresh.py`'s learning), PEG untested (no historical source, Pham tested technical-half-only), an invented exit rule since none exists live for equities (SL-or-T1-or-20-trading-day-timeout — verified only 5.7% of trades actually hit the timeout, so this isn't diluting the result), trade-return-based Sharpe (not NAV-based), and a chronologically-compounded max-drawdown that overstates real risk (doesn't model spreading capital across concurrent positions). **First real run's result (49 stocks, 3y, 1,486 trades) is a genuine "not yet a clear go" signal, not a rubber stamp**: overall 49.1% win rate / Sharpe~2.05 / max DD -54.14%, three of four voters (Inna 49.0%, Cianni 45.2%, Unger 48.7%) are statistically indistinguishable from a coin flip and **Cianni is net-negative (-0.06% avg)**, only Pham shows real edge (55.8% win rate, best avg return) — and the whole system's per-trade returns look weak next to a plain 12.26%/yr buy-and-hold of the same universe. Don't reflexively re-run this expecting a different verdict without changing the strategy logic or the universe — the data currently argues Pham should be weighted up and Cianni questioned, well before `voter_weights_refresh.py` would ever accumulate enough real trades to learn the same thing itself.
- **F&O Gate 2 (`nse-trading-bot/backtest_fo.py`) built 08-Jul-2026** — the equity backtest's counterpart, covering what it doesn't: index F&O (NIFTY50/BANKNIFTY 3-factor consensus, `refresh_fo_cloud.py`) + stock F&O (18 tracked stocks, single-leg AND spread, `stock_fo_refresh.py`). Same rigor: `_signals`/`_premium_estimate`/`_atm` imported directly (zero drift on the decision/pricing math); the only backtest-specific code is the walk-forward indicator series and `_next_monthly_expiry_asof()` (a date-parameterized twin of `_next_monthly_expiry()`'s calendar math, since that function hardcodes `date.today()` — verified byte-identical output against the real function). Cross-checked the walk-forward indicators against production `_get_indicators()` — matched exactly when fed identical data (a same-day live-fetch timing artifact caused a *false* mismatch on first check, resolved by re-testing on a shared dataframe, not a logic bug). **Unlike equity, F&O has a REAL live exit rule already** (`trade_brain.py`'s target2>target1>stop_loss>expired>signal_flip priority) — reused as-is, no invented convention needed for single-leg trades. Spreads settle only at expiry vs breakeven (same coarse convention as `recommendation_tracker.py`'s `fo_stock_spread` scoring — no two-leg time-value modeling). Sharpe/max-drawdown use the identical method as `backtest.py` for direct comparability.
  **First real run (3y, 1,616 trades total)** — genuinely mixed, worth reading carefully, not averaging away: **Index F&O shows real edge** (218 trades, 55.5% win rate, Sharpe 4.18 — NIFTY50 alone 58.2%/Sharpe 3.44, BANKNIFTY 52.8%/Sharpe 2.47 — meaningfully better than equity's near-coin-flip result). **Stock F&O single-leg (max-conviction 6/6 votes) is sub-coin-flip** (804 trades, 39.2% win rate). **Stock spreads (4-5 votes) also sub-coin-flip on win rate** (594 trades, 38.6%) **but show a large positive average return (116.07%)** — a real, structural property of capped-risk debit spreads (small net-debit base means an uncapped-until-width win dwarfs a capped -100% loss in % terms), not an error; read win-rate AND Sharpe/maxDD together for these, never avg_return_pct alone. **Counter-intuitive finding worth acting on**: by vote-count, 6/6 "max conviction" stock trades had the WORST win rate (39.8%) and lowest avg return (11.0%) of any bucket — WORSE than the 4-5 vote spread bucket (41.9%/45.3% win, 92.84%/110.85% avg) — directly contradicting the "prefer higher vote counts" intuition baked into `expert_gate.py`'s `MIN_VOTES_FOR_ENTRY` tunable comment. Don't raise that threshold expecting it to help stock F&O without accounting for this. Max-drawdown reads ~-100% almost everywhere — expected given the compounding-overstates-risk caveat (options frequently do lose 100% of premium; chronologically compounding 800+ trades makes hitting that trough at least once nearly inevitable) — this is NOT "you'd lose all your capital," see the disclosed method caveat before reacting to that number alone.
